# 🚀 WG Hope Bot

**WG Hope Bot** — Telegram-бот для управления WireGuard-доступами + веб-мониторинг пользователей, конфигов и uplink-интерфейсов.

Проект собран для production-эксплуатации: роли, лимиты, аудит действий, регионы выхода в интернет, healthcheck каналов и автоматический failover.

## ✨ Что умеет проект

- 👥 Роли: `super_owner`, `admin`, `user`, `pending`, `banned`
- 🔐 Выдача VPN-доступа через Telegram (`.conf` + QR)
- 🧩 Лимиты, рассылки, логи, статистика, модерация заявок
- 🌍 Регионы выхода для каждого конфига:
  - `moscow` -> `eth0`
  - `latvia` -> `aw-lv` (default)
  - `amsterdam` -> `aw-am`
- 🔁 Policy routing по регионам и автоматический failover на системный uplink при падении VPN-uplink
- 🛠️ Админка серверов в боте:
  - интерфейсы/регионы
  - замена uplink-конфига
  - установка default-региона
  - удаление интерфейсов/регионов кнопками
- 📊 Веб-мониторинг:
  - пользователи/конфиги/события
  - вкладка `Серверы` (uplink-интерфейсы, uptime, скорость, ping, события состояния)
  - фильтры, графики, периоды
  - отображение `Время сервера` на всех страницах

## 🆕 Актуальный релиз

- `v1.3.3` — production hotfix производительности и стабильности Telegram-бота + веб-мониторинга:
  - устранены длительные задержки при открытии веб-морды и при работе бота под нагрузкой;
  - оптимизированы SQL-запросы мониторинга:
    - тяжёлые realtime-вычисления скоростей вынесены в лёгкий агрегат `wg_peer_rates`;
    - данные «последнего состояния» читаются из `wg_peer_state` вместо сканов `wg_peer_samples`;
    - снижено число N+1 запросов на dashboard (handshake/top destinations считаются батчем).
  - усилена устойчивость SQLite в проде:
    - включён `WAL`-режим;
    - увеличены таймауты ожидания блокировок (`busy_timeout`);
    - уменьшен риск `sqlite3.OperationalError: database is locked` при одновременной работе бота и веба.
  - снижена нагрузка на I/O и CPU при активном мониторинге без потери текущего функционала.
- `v1.3.2` — UX/стабильность админки и доступа для заблокированных:
  - исправлен сценарий `Админка -> Сообщения`:
    - кнопки групп (`Ожидают`, `Одобренные`, `Забаненные`, `Все`) снова корректно ведут к выбору получателей;
    - `Добавить chat_id` и `Далее к тексту` работают стабильно даже при потере контекста диалога (после рестарта/сброса состояния).
  - добавлено отдельное меню для роли `banned`:
    - доступны кнопки `Статус` и `Вопросы / Поддержка`;
    - убрана ситуация, когда у заблокированных отображалось обычное меню с недоступными действиями.
  - добавлено сохранение причины блокировки:
    - новая таблица `user_blocks` в БД;
    - причина и время блокировки записываются при бане и используются в ответе по статусу.
  - улучшен ответ `Статус` для заблокированных:
    - дата блокировки показывается в формате `dd-mm-YYYY HH:MM:SS`;
    - время выводится в локальной TZ сервера (без отставания UTC-формата).
- `v1.3.1` — hotfix стабильности метрик после reboot в веб-мониторинге:
  - устранён сброс `RX/TX всего` на главной и в карточках пользователей после перезагрузки сервера;
  - добавлена накопительная таблица `wg_peer_totals`:
    - totals считаются инкрементально и переживают reboot;
    - добавлен автоматический backfill из исторических `wg_peer_samples` для уже работающих инсталляций.
  - устранён сброс `RX/TX всего` во вкладке `Серверы` после перезагрузки;
  - добавлена накопительная таблица `uplink_totals`:
    - totals считаются по uplink-интерфейсам (`aw-*`, `eth0`) независимо от reset сырых счётчиков интерфейсов;
    - добавлен автоматический backfill из `uplink_samples`.
  - обновлены источники данных в мониторинге:
    - `dashboard` + realtime API пользователя используют накопительные `wg_peer_totals`;
    - вкладка `Серверы` + realtime API серверов используют накопительные `uplink_totals`.
  - результат:
    - после reboot графики продолжают жить как раньше;
    - поля `RX/TX всего` больше не обнуляются из-за обнуления kernel/wg counters.
- `v1.3.0` — стабильность после reboot, расширенный UX админки и real-time мониторинг:
  - добавлен автоматический post-reboot healthcheck с отчётом в Telegram админам:
    - проверка default route, managed `ip rule`, `wg0`, `wg-hope-bot`, `wg-hope-monitor`, SSH;
    - проверка uplink-интерфейсов/регионов, авто-очистка устаревших health-записей;

    - ручной запуск теста из бота: `Админка -> Логи -> Ручной тест`.
  - усилена безопасность добавления uplink:

    - `Синхронизировать профили` теперь даёт подробный отчёт по ошибкам (chat_id, username, причина).
  - веб-морда переведена на real-time обновления:

    - live обновление графиков пользователя и интерфейсов;
    - добавлены поиск/фильтры по пользователям и расширенные карточки статистики;
    - на главной добавлены блоки `Новые заявки` и `Нагрузка сервера` (CPU/RAM/Disk/Load/Uptime);
    - исправлена обработка legacy-времени в логах и приведён вывод времени к локальной TZ сервера;
    - в `Топ регионов` учитываются даже регионы с нулём конфигов (новые регионы видны сразу).

## 🗂️ Структура

- `vpn_bot/` — Telegram-бот
- `monitor/` — FastAPI веб-мониторинг
- `chat/` — история чатов пользователей
- `clients/` — клиентские конфиги/QR/метаданные
- `deploy/systemd/` — шаблоны systemd-юнитов
- `deploy/scripts/` — вспомогательные скрипты развёртывания
- `deploy/SERVER_SETUP_PROD_v1.2.0.md` — пошаговая серверная настройка uplink/регионов/failover
- `CHANGELOG.md` — история изменений

## ⚙️ Требования

- Linux-сервер
- Python 3.11+
- `wg`, `wg-quick` (WireGuard)
- `iproute2`, `iptables`
- `conntrack` (для аналитики направлений трафика в мониторинге)
- Для AmneziaWG uplink-ов: `awg`, `awg-quick` и соответствующие systemd-сервисы (например `amnezia-awg@aw-lv.service`)

## ▶️ Быстрый запуск (локально)

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
# заполнить BOT_TOKEN, SUPER_OWNER_CHAT_ID, SERVER_ENDPOINT, SERVER_PUBLIC_KEY
python -m vpn_bot.main
```

## 🌐 Запуск мониторинга

```bash
uvicorn monitor.app:app --host 0.0.0.0 --port 8080
```

Вход по умолчанию: `admin / admin` (сразу сменить в `Настройки`).

## 🧪 Production-развёртывание (рекомендовано)

1. Клонировать проект в `/opt/wg-hope-bot`.
2. Создать venv и установить зависимости.
3. Заполнить `.env` (см. `.env.example`).
4. Скопировать systemd-юниты из `deploy/systemd/`.
5. Включить сервисы:

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now wg-hope-bot.service
sudo systemctl enable --now wg-hope-monitor.service
```

6. Проверить:

```bash
systemctl is-active wg-hope-bot.service
systemctl is-active wg-hope-monitor.service
```

## 🌍 Регионы и uplink routing

- Каждый клиентский конфиг хранит поле `region`.
- `sync_client_egress_routes()` строит policy routing и iptables для выхода через интерфейс региона.
- При падении VPN uplink трафик клиентов соответствующих регионов временно уходит через системный uplink (обычно `eth0`).
- После восстановления интерфейса маршрутизация возвращается автоматически.

  - `WG_INTERFACE`, `WG_CONF`

  - `PROJECT_DIR`, `CLIENTS_DIR`, `DB_PATH`, `CHAT_DIR`

  - `DEFAULT_USER_LIMIT`, `ADMIN_LIMIT`
- Веб-мониторинг:
  - `WEB_SECRET`, `WEB_PORT`
  - `MONITOR_POLL_SEC`, `MONITOR_KEEP_DAYS`, `MONITOR_DNS_CACHE_DAYS`

  - `UPLINK_HEALTHCHECK_INTERVAL_SEC`
  - `UPLINK_HANDSHAKE_STALE_SEC`
  - `UPLINK_DOWN_CONFIRM_COUNT`
  - `UPLINK_UP_CONFIRM_COUNT`
